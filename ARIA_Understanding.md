## 1. The Core Vision — What ARIA Actually Is

ARIA is not a chatbot. It is not a content tool. It is not a dashboard. ARIA is a **one-input marketing operating system** that replaces every marketing tool a business currently pays for, and executes the work automatically.

The product thesis is simple: people do not come to ARIA for a "campaign." They come because they need something right now — a video, a landing page, an email sequence, a competitor analysis. ARIA reads that intent, routes it to the correct agents, and delivers a complete, professional output. No menus. No forms. No navigation. One input box.

> "ARIA is the product. Everything else is infrastructure running underneath."

The competitive window is real and time-limited. Mega AI raised $11.5M from a16z on March 10, 2026 at $299/month with humans in the loop. ARIA delivers the same capability — fully automated — at $49/month. That gap will not last 12–18 months.

## 2. How ARIA Works — The Three Checks

Before building anything, ARIA runs three checks on every single request:

| Check | What ARIA Does |
|---|---|
| **Who are you?** | Loads Brand Kit — tone, colors, audience, competitors. If none exists, asks ONE question to begin building it. |
| **What have you built?** | Checks folder history. References past campaigns to inform the current build. Never starts from zero for returning users. |
| **What do you need now?** | Reads intent from the message. Routes to the correct module. Builds immediately. |

## 3. The Three Entry Points

ARIA detects which mode the user is in from their first message. No selection required.

**Entry Point 1 — Full Campaign Builder.** Triggered when the user wants to launch something new or needs the full marketing stack. The flow is: (1) user types one sentence → ARIA searches the web for the existing brand → returns a 3–4 bullet strategy summary → "Build Campaign" button appears; (2) a dynamic checklist pre-populated from domain intelligence appears — not hardcoded, different every time; (3) a progress screen shows each agent completing by name, then the full campaign folder opens automatically.

**Entry Point 2 — Existing Brand.** Triggered when the user has a website or existing brand. ARIA reads the brand from the URL or upload, loads the Brand Kit automatically, shows only what is missing, and never rebuilds what already exists.

**Entry Point 3 — Specific Task.** Triggered when the user needs one thing fast. Zero questions. Zero checklist. Immediate execution. Saves to a folder automatically. Option to expand into a full campaign after delivery.

## 7. Real Data Integration — DataForSEO

**The core problem the current build has:** Website analysis currently passes a URL to the AI and asks it to estimate numbers. The AI hallucinates traffic, domain authority scores, and keyword data from its training. A user who runs this on their own website will immediately know the numbers are wrong. That single moment destroys all trust.

**The fix:** Use real data APIs. DataForSEO costs approximately $0.001 per call. At 1,000 users running 10 campaigns per month, DataForSEO costs $380/month total against $75,950/month in revenue — 0.5% of revenue.

### 7.1 SEO Data Honesty Rule — Interim Behavior Before DataForSEO Is Wired

**This rule governs what ARIA shows until DataForSEO is integrated.** The current build uses cheerio + axios to fetch real page content (on-page text, meta tags, headings, links). This is real data. However, cheerio cannot provide traffic estimates, domain authority scores, backlink counts, or keyword rankings — those require DataForSEO.

**The interim rule is strict:**
- Any metric that requires DataForSEO (DA score, monthly traffic, keyword volume, backlink count, SERP position) must be labeled **"AI Estimate — connect DataForSEO for verified data"** in the UI.
- Alternatively, strip those fields entirely from the output card until DataForSEO is wired.
- Under no circumstances should ARIA display a specific number (e.g., "DA: 42" or "Traffic: 18,500/mo") that was generated by the LLM from training data. Users will check these numbers against real tools and the trust damage is permanent.
- On-page content (copy analysis, meta tags, headings, internal link structure, page speed via Google PageSpeed API) can be shown as real data — cheerio provides this accurately.

### 7.3 Cheerio Ceiling — What It Can and Cannot Do

**cheerio + axios solves half the problem, not all of it.** This is a critical distinction for any developer reading this document.

What cheerio provides (real, accurate):
- Full page text content
- All meta tags (title, description, og:image, og:title, canonical)
- All headings (H1–H6) and their text
- Internal and external link structure
- Image alt tags
- Structured data / JSON-LD blocks if present on page
- Technology hints from script src attributes

What cheerio cannot provide (requires DataForSEO or equivalent):
- Domain Authority (DA) score
- Monthly organic traffic estimate
- Backlink count and profile
- Keyword rankings in Google
- Keyword search volumes
- Competitor keyword gap data
- Core Web Vitals (use Google PageSpeed API for this instead)

A developer reading this document should understand: cheerio is the foundation for on-page intelligence. DataForSEO is the layer that adds off-page and competitive intelligence. Both are required for the full Website Intelligence Analyzer feature.

## 8. Result Card Presentation Standards — JSON-First Architecture

### 8.1 The Architectural Decision

**The root cause of the current paragraph-dump problem:** Agents currently return text strings. The UI wraps those strings in a card shell. This produces formatted paragraphs inside a card — not a structured result card.

**The required architecture:** Every agent must return **structured JSON**, not prose. The UI renders from JSON fields. This is the only way to produce metric tiles, sortable tables, color-coded scores, and action buttons — none of which can be derived from a text string.

The implementation pattern for every agent:

```typescript
// ❌ Current (wrong): agent returns a string
return { message: "Your domain authority is around 42 and you get about 18,500 visitors per month..." };

// ✅ Required: agent returns structured JSON
return {
  domainScore: 42,
  monthlyTraffic: 18500,
  trafficSource: { organic: 62, paid: 18, social: 12, direct: 8 },
  topKeyword: "marketing automation software",
  strengths: ["Strong blog content", "Good page speed", "Active social presence"],
  gaps: ["No backlink strategy", "Missing schema markup", "Thin product pages"],
  opportunity: "Competitor X ranks for 340 keywords you don't — content gap is your fastest win"
};
```

The UI component then reads `result.domainScore`, `result.strengths`, etc. and renders each field into its designated visual slot. The LLM is instructed via `response_format: { type: "json_schema" }` to return structured output. This is already supported by the `invokeLLM` helper.

**This architectural change must be applied to every agent in the DAG before Pass 1 ships.** Without it, the campaign folder will display paragraphs, not structured cards.

### 8.2 Result Card Specs by Output Type

Every output type needs a dedicated card component. The rule: **metric first, number before explanation, red/amber/green status on every number, one clear action per card, expandable detail, download always available.**

| Output Type | Card Shows | Presentation |
|---|---|---|
| Website Analysis | Domain score badge (0–100) + 4 metric tiles (Traffic / DA / Organic % / Top keyword) + Top 3 Strengths + Top 3 Gaps + 1 Opportunity callout | Report card layout, color-coded scores, expandable sections |
| Competitor Ad Spy | Grid of competitor ad creatives (thumbnail + copy snippet), filter by platform, age of ad | Visual ad gallery, not a text list |
| Keyword Gap | Table: Keyword / Competitor Rank / Your Rank / Volume / Difficulty / Opportunity Score | Sortable table, green = easy win, red = hard battle |
| Keyword Matrix | Top 20 keywords: keyword, monthly volume, difficulty (1–100), current rank, opportunity score | Sortable table, download as .xlsx, one-click "Create content for this keyword" |
| Technical SEO Audit | Score /100 at top, issues list with severity badges (Critical / Warning / Info), one-line fix per issue | Accordion list, critical issues always expanded, copy fix text button |
| Campaign Strategy | Collapsible sections: Positioning / Audience / Competitors / Messaging / Channels / 90-day plan | Document layout, download as .docx |
| Social Posts | Platform-native preview for each post (Instagram square, LinkedIn post, Twitter card) | Visual platform preview, looks like the real platform, schedule button per post |
| Ad Creative | Ad mockup: headline + body + CTA button + placeholder image, platform badge, predicted CTR | Ad mockup card, edit headline/copy inline, publish to platform button |
| Email Sequence | Numbered email cards: subject line + preview text + send timing, expand to full email | Email sequence cards, expandable, Send via Resend button per email |
| Campaign Score | Large score badge /100, sub-scores (Content / Reach / Engagement / Conversion), one recommendation | Score ring graphic at top of folder, always visible |

### 9.7 ARIA_MODEL Config Constant

A single `ARIA_MODEL` constant must be added to `server/_core/llm-provider.ts` (or equivalent config file). Its purpose: **the AI model can be swapped across the entire codebase in one line without hunting through files.**

Currently, if the model needs to change (e.g., from one provider to another, or to a newer version), every agent file that calls `invokeLLM` would need to be updated individually. With `ARIA_MODEL` as a single exported constant, all agents import it and the change propagates everywhere instantly.

```typescript
// server/_core/llm-provider.ts
export const ARIA_MODEL = process.env.ARIA_MODEL ?? "claude-sonnet-4-5";

// Every agent uses it:
import { ARIA_MODEL } from "../_core/llm-provider";
const response = await invokeLLM({ model: ARIA_MODEL, messages: [...] });
```

This is a one-line change with significant operational value — especially as the model landscape evolves rapidly.

### 9.8 DAG Latency — Expected Behavior, Not a Bug

**The DAG has an 18–20 second minimum latency on complex requests.** This is not a bug. It is the sum of LLM API latency across the pipeline:

| Stage | Typical Duration |
|---|---|
| StrategyAgent (intent + planning) | ~5 seconds |
| Parallel agent execution (content, SEO, email, social, ads) | ~8 seconds (parallel, not sequential) |
| ReviewAgent (quality check + assembly) | ~5 seconds |
| **Total minimum** | **~18–20 seconds** |

A developer or tester reading this document should not file a bug report when the DAG takes 18–20 seconds. The ThinkingIndicator with live agent progress pills was added in Phase 17 specifically to make this latency feel intentional and transparent rather than broken. If a request takes longer than 45 seconds, that is a timeout issue worth investigating. Under 45 seconds is expected behavior.
